# Контекст для ИИ: правила работы с документацией

Этот репозиторий — источник markdown-документации. Статические страницы сайта-базы знаний генерируются напрямую из файловой структуры каталога `storage/`. Здесь нет кода приложения — только контент.

## Как устроено хранилище

```
storage/
  <lang>/              # код языка: ru, en, ...
    <path>/<slug>.md   # одна статья = один .md-файл
```

- Корень контента — `storage/`.
- Первый уровень — язык (`ru`, `en`). При генерации страниц языковой префикс **отбрасывается**.
- Подкаталоги задают иерархию разделов.
- Имя файла без расширения — это slug последнего сегмента URL.

### Соответствие пути файла и URL

| Файл | URL страницы |
|---|---|
| `storage/ru/getting-started.md` | `/getting-started` |
| `storage/ru/glossary.md` | `/glossary` |
| `storage/ru/projects/aurora/passport.md` | `/projects/aurora/passport` |
| `storage/ru/projects/aurora/architecture.md` | `/projects/aurora/architecture` |

Из этого следует главное правило именования: **имя файла должно совпадать с желаемым slug'ом**, а не с русским заголовком. Только латиница в нижнем регистре, слова через `-`.

## Правила для ссылок

**Ссылки всегда корневые (root-relative), без расширения и без языкового префикса.**

- Правильно: `[Паспорт проекта](/projects/aurora/passport)`
- Неправильно: `[Паспорт](../passport.md)` — относительный путь, сломается при переносе файла
- Неправильно: `[Паспорт](/ru/projects/aurora/passport)` — языковой префикс не входит в URL
- Неправильно: `[Паспорт](/projects/aurora/passport.md)` — расширение `.md` в URL не нужно

Якоря на заголовки внутри страницы пишутся через `#` в кириллице в нижнем регистре, слова через `-`:
`[Цели и задачи](/projects/aurora/technical-specification#цели-и-задачи-проекта)`.

Внешние ссылки — обычные абсолютные URL (`https://...`).

## Frontmatter

YAML-шапка в начале файла — опциональна. Поддерживается как минимум:

```yaml
---
title: Wiki
---
```

`title` переопределяет заголовок страницы (иначе берётся первый `# H1` из тела). Не дублируй `title` во frontmatter и первый `# Заголовок` в теле, если они одинаковые — оставь что-то одно.

## Структура статьи

- Ровно один `# H1` на файл (если нет `title` во frontmatter).
- Разделы — `##`, подразделы — `###`. Не прыгай через уровни.
- Списки через `-`. Нумерованные — только когда порядок важен (шаги, этапы).
- Код — в тройных бэктиках с указанием языка.

## При создании новой статьи

1. Выбери язык: положи файл в `storage/<lang>/...`.
2. Подбери путь так, чтобы он читался как URL: `storage/ru/projects/aurora/roles.md` → `/projects/aurora/roles`.
3. Slug — только латиница, цифры и `-`. Никаких пробелов, кириллицы, `_` или CamelCase в именах файлов.
4. Проверь входящие ссылки: если статья упоминается из других мест, их URL мог поменяться — поправь.

## При переименовании или перемещении статьи

Меняется URL → все ссылки на этот URL в других `.md`-файлах **ломаются молча** (сборка страницы этого не увидит). Перед переименованием:

1. Найди все упоминания старого URL по всему `storage/` (`grep` по точной строке `/старый/путь`).
2. Обнови их на новый URL в том же коммите.
3. Если старый путь был публичным — обсуди с пользователем, нужен ли редирект/заглушка.

## Стиль

- Язык статьи = язык каталога (`storage/ru/` → по-русски, `storage/en/` → по-английски). Не смешивай.
- Имена файлов и slug'и — латиницей, даже в русскоязычных разделах.
- Термины в глоссарии ([/glossary](storage/ru/glossary.md)) — источник правды для наименований сущностей. Если вводишь новый термин в статье — добавь его и в глоссарий.

## Чего НЕ делать

- Не класть файлы вне `storage/` с намерением, что они станут страницами — они не попадут в сайт.
- Не использовать относительные ссылки (`../foo.md`, `./bar.md`) между статьями.
- Не оставлять `.md` и языковой префикс в URL ссылок.
- Не создавать дубли статей на одном языке под разными slug'ами — одна тема = один канонический файл.
- Не писать README/CONTRIBUTING внутри `storage/` — они попадут в сайт как страницы. Служебные документы должны жить в корне репозитория.
