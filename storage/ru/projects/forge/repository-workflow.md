# Работа с репозиторием Forge

## Branching-стратегия: trunk-based

Forge придерживается **trunk-based development**. Это сознательный выбор: продукт развивается быстро, и долгоживущие feature-ветки приводят к рассинхронизации между разработчиками. Вместо них — короткие ветки от `main`, частые мерджи и фича-флаги.

```
main                ── единственная долгоживущая ветка, всегда деплоится
  ├─ FORGE-1234-...  ── короткая ветка (≤ 2 дней), мерджится в main
  └─ FORGE-2345-...
```

Правила:

- ветки от `main`, имя — `<JIRA-ID>-<краткое-описание>`;
- срок жизни ветки — не больше двух рабочих дней. Долгие задачи декомпозируются;
- незавершённые фичи защищены **фича-флагами** (LaunchDarkly), мерджи в `main` идут даже когда фича ещё не готова к включению (см. [Процессы](/projects/forge/processes));
- прямой push в `main` запрещён настройкой репозитория.

## Конвенция коммитов

Используем **Conventional Commits**:

```
feat(editor): support nested groups in field tree
fix(runtime): handle empty schema gracefully
chore(deps): bump zustand to 4.5.2
```

Допустимые типы: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `chore`, `build`, `ci`. Scope — имя модуля (`editor`, `runtime`, `core`, `routes`, `shared`).

Сообщения должны быть на английском (стандарт студии). При мердже PR через `Squash and merge` собирается финальное сообщение по этому шаблону.

## Pull Request

### Создание

PR из feature-ветки → `main`. Шаблон PR:

```markdown
## Что
Краткое описание изменений.

## Зачем
Ссылка на Jira, контекст.

## Фича-флаг
Имя флага (если применимо), default-значение.

## Как проверить
Шаги. Если за флагом — как его включить локально.

## Чек-лист
- [ ] CI зелёный
- [ ] Тесты добавлены
- [ ] Бандл-чек пройден
- [ ] Не сломаны существующие сохранённые схемы
- [ ] Скриншоты UI приложены
```

### Размер

Цель — PR до 300 строк. Это короче, чем в Pulse, потому что trunk-based стимулирует мелкие изменения.

### Ревью

- минимум один апрув от Senior ([Марина](/team#марина));
- архитектурные изменения требуют апрува Dev Lead ([Андрей](/team#андрей));
- срок ответа на запрос ревью — рабочий день, по возможности — в течение 4 часов;
- PR от [Дарьи](/team#дарья) (Junior) обязательно ревьюит [Марина](/team#марина) с менторскими комментариями.

### Чек-лист ревьюера

1. Соответствие [Код-стайл](/projects/forge/code-style).
2. Архитектурная границы (editor / runtime / core) не нарушены.
3. Если затронуты core-операции — добавлены или обновлены юнит-тесты.
4. Если меняется модель схемы — есть миграция и unit-тест миграции.
5. Бандл-чек: рантайм ≤ 180 КБ, редактор ≤ 480 КБ (см. [Архитектура приложения](/projects/forge/architecture)).
6. Фича за флагом, если она ещё не должна быть видна пользователям.
7. UI-проверка на одном из пилотных профилей staging.

## CI

GitHub Actions, на каждый PR в `main`:

| Шаг | Среднее время |
|---|---|
| `npm ci` (с кэшем) | 50 сек |
| ESLint | 30 сек |
| Type-check | 1 мин |
| Vitest (с покрытием) | 2.5 мин |
| Playwright smoke | 4 мин |
| Bundle size check | 30 сек |
| Vercel preview deploy | 1 мин |
| Chromatic visual regression | 2 мин |

Любой красный шаг блокирует мердж.

## Релизы

Trunk-based в Forge — это **continuous delivery**: каждый мердж в `main` автоматически уходит на production-окружение, фичи скрываются за флагами.

Включение флага — отдельная операция, инициирует PO. Включение пишется в `releases/`-журнал в Confluence с указанием:

- имя флага;
- скоуп (все клиенты / пилотные / отдельный клиент);
- дата;
- ссылка на PR(ы) с кодом фичи;
- ссылки на staging-проверку.

## Hotfix

Hotfix — это просто короткий PR в `main`. Отдельной ветки `hotfix/*` нет. Если фикс критичен и нужно ускорить мердж, это указывается в названии PR (`hotfix:`) и обсуждается в `#forge-team`.

> ⚠️ Внимание: даже hotfix должен пройти полный CI. Игнорировать падающие шаги — запрещено. При необходимости — Марина и Андрей могут принять решение о временном `skip` конкретного шага, но это документируется.

## Откаты

Откат фичи — это, в первую очередь, выключение флага. Это безопаснее любого revert'а в коде.

Если фича уже глобально включена, и нужно её свернуть, делается revert-PR через `git revert` целевого коммита, с обязательным мерджем через тот же CI.
