# Архитектура приложения Forge

## Подход

Forge построен по подходу **«ядро + редактор + рантайм»**. Вместо классического FSD (как в [Pulse](/projects/pulse/architecture)), здесь выделены три независимых модуля, каждый со своим внутренним делением.

```
src/
  app/                  # Next.js App Router: layout, провайдеры
  modules/
    core/               # ядро: типы схемы, валидация, сериализация
    editor/             # конструктор форм (DnD, дерево полей)
    runtime/            # исполнение формы у конечного пользователя
    routes/             # маршруты согласования
  shared/
    ui/                 # общая UI-библиотека (см. ui-library)
    api/                # HTTP-клиент и интеграции
    lib/                # утилиты, хуки
  pages-content/        # MDX-документация для конструкторов
```

## Модули

### Core

Ядро — единственный источник правды о модели формы. Здесь живут:

- TypeScript-типы схемы (`FormSchema`, `Section`, `Field`, `ValidationRule`);
- генерация и валидация JSON Schema (см. [Используемые технологии](/projects/forge/tech-stack));
- сериализация и десериализация (включая backward compatibility);
- helpers-операции над схемой (добавить поле, переместить секцию, дублировать группу).

Ни Editor, ни Runtime не содержат логики мутации схемы напрямую — только через core-операции. Это делает поведение предсказуемым в обеих средах.

### Editor

Редактор — клиентское приложение, целиком работающее с Zustand-стором. Сторы:

- `useEditorStore` — текущая схема, история (undo/redo), выделение;
- `useDndStore` — состояние перетаскивания;
- `useUiStore` — режимы редактора (preview, schema, settings).

Рендеринг полей в редакторе — отдельный набор компонентов `Field*Editor`, в отличие от рантайма. Они показывают границы, ручки изменения размера, контекстное меню.

### Runtime

Рантайм — то, что видит конечный пользователь, заполняющий форму. Получает схему, делает её гидрацию, валидирует ввод по предсобранному Ajv-валидатору. Никакой логики редактирования.

В рантайме намеренно отсутствует Immer и большая часть зависимостей редактора — это сделано, чтобы держать бандл рантайма меньше 180 КБ (gzipped, цель). Текущий размер — 165 КБ.

### Routes (модуль маршрутов согласования)

В разработке. Отвечает за визуальный конструктор маршрутов и runtime-выполнение шагов согласования. Опирается на core (для модели данных формы) и имеет собственное ядро описания маршрута.

## Управление состоянием

| Тип | Решение |
|---|---|
| Серверное | Direct fetch + Next.js cache + SWR (там, где нужен ревалидейт) |
| Глобальное (редактор) | Zustand + Immer middleware |
| Глобальное (рантайм) | Zustand без Immer (плоская модель ответов) |
| Локальное | `useState` |

SWR используется вместо React Query, потому что естественно интегрируется с Next.js Data Fetching и проще в SSR-контексте.

## Производительность

Известное ограничение — деградация редактора при формах больше 200 полей. Профайлинг показал, что узкое место — глубокая нормализация в Immer. Принятый RFC: переход на нормализованную модель (`{ byId, allIds }`) внутри стора, с миграцией постепенно. Сроки — Q3 (см. [Беклог и roadmap](/projects/forge/backlog-roadmap)).

Цель по бандлу: редактор ≤ 480 КБ gzipped, рантайм ≤ 180 КБ. Проверка в CI.

## Server Components vs Client

| Раздел | Тип |
|---|---|
| Лендинг и галерея шаблонов | RSC, ISR с ревалидейтом 5 минут |
| Документация (MDX) | RSC, статическая генерация |
| Редактор (`/editor/[id]`) | Client (`'use client'`) |
| Рантайм формы (`/forms/[publicId]`) | Hybrid: shell — RSC, тело — Client |
| Личный кабинет конструктора | Client |

## Аутентификация и SSR

Сессии — JWT в `httpOnly` cookie. Серверный код Next.js читает cookie через `next/headers` и проксирует токен в `Authorization` для API-запросов. Подробнее — в [API-взаимодействие](/projects/forge/api-integration).

## RFC-процесс

Любое изменение, затрагивающее core-типы или контракт между модулями, требует RFC. Шаблон:

1. Контекст и проблема.
2. Альтернативы.
3. Предложение и влияние на core / editor / runtime / routes.
4. Совместимость со старыми сохранёнными схемами (миграция или отказ).

RFC апрувит [Марина](/team#марина) и [Андрей](/team#андрей). Особое внимание — обратной совместимости: пользовательские схемы хранятся годами и должны открываться без ручного вмешательства.

> ⚠️ Внимание: любое изменение модели схемы должно сопровождаться миграцией старых документов. Запрещено вводить новые обязательные поля без default-значения.
