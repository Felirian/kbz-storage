# Беклог и roadmap Forge

## Принципы приоритизации

Forge использует **MoSCoW** для квартального планирования и **RICE** — для отбора задач внутри категорий.

- **Must** — без этого квартал считается провальным;
- **Should** — высокая ценность, реализуем при свободной ёмкости;
- **Could** — приятные улучшения, идут только если остаётся время;
- **Won't (this quarter)** — явно отложено, чтобы не возвращаться к обсуждению.

Внутри Must и Should задачи упорядочиваются по RICE (формула — см. описание в [Беклог и roadmap Pulse](/projects/pulse/backlog-roadmap), команды используют её одинаково).

Особенность Forge — наличие отдельного источника приоритетов: `#forge-pilot-feedback`. Каждое второе планирование Наталья пересматривает беклог с учётом фидбека пилотных клиентов.

## Текущий квартал (Q2 2025)

### Must

| Инициатива | RICE | Статус |
|---|---|---|
| Шаблонная галерея форм (16 типовых процессов) | 31.5 | в работе |
| Упрощение онбординга конструктора (tour + first-form wizard) | 24.0 | дизайн |
| Исправление перформанса редактора при > 200 полях | 18.2 | RFC принят |
| Импорт схемы формы из Excel | 14.7 | беклог |

### Should

| Инициатива | RICE |
|---|---|
| Версионирование схем формы | 11.4 |
| Превью маршрута согласования в реальном времени | 9.8 |
| Расширение набора полей (rating, signature) | 7.3 |

### Could

- Тёмная тема (запрос от одного пилотного клиента);
- Расширенные хоткеи для конструктора;
- Экспорт схемы в JSON Schema файл.

### Won't (this quarter)

- Мобильный конструктор — недостаточно сигналов о потребности;
- Маркетплейс шаблонов от сообщества — слишком рано без платной модели.

## Roadmap

### Q2 2025 (текущий)

- галерея шаблонов в production;
- метрика «время сборки первой формы» сокращена до целевых 25 минут;
- релиз новой архитектуры состояния редактора (см. [Архитектура приложения](/projects/forge/architecture)).

### Q3 2025

- модуль «Маршруты согласования» в production;
- роли и разрешения для конструкторов внутри организации клиента;
- интеграция с корпоративным e-mail (отправка уведомлений по событиям маршрута).

### Q4 2025

- переход на платную модель (тарифы, биллинг через Stripe);
- расширенная аналитика для PO клиента (дашборд по форме);
- публичный API чтения данных форм для интеграции с BI клиента.

## Критерии перевода задачи в спринт

Чтобы задача попала в спринт, должно быть выполнено всё:

- категория MoSCoW проставлена и не Won't;
- описание в Jira содержит «Контекст / Гипотеза / Acceptance»;
- дизайн утверждён Викторией (если требует UI);
- если архитектурно неочевидно — есть RFC и апрув от Марины и Андрея;
- задача декомпозирована до задач не более 2 рабочих дней.

## Технический долг

Forge сознательно копит технический долг в обмен на скорость, но раз в квартал команда фиксирует «потолок». Текущий список:

1. Нормализация модели схемы формы — сейчас глубоко вложенный объект через Immer, для больших форм это начинает ощущаться.
2. Замена внутренней реализации поля «Таблица» на готовый компонент TanStack Table.
3. Покрытие core-движка интеграционными тестами (сейчас покрыты только юнит-тесты).
4. Удаление legacy-эндпоинтов API v0.

> ⚠️ Внимание: пункт 1 заблокирован до релиза модуля маршрутов — рефакторинг во время разработки маршрутов слишком рискован. Ответственный за пересмотр: [Марина](/team#марина). Приоритизация — Q3.
