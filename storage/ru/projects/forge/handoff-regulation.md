# Регламент передачи макетов Forge

## Цель

Forge — продуктовая разработка с короткими спринтами, поэтому регламент хендоффа выстроен на минимум формальностей, но с обязательными проверками, без которых задача не уходит в работу.

## Поток передачи

```
Figma working file
   ↓ обсуждение с разработчиком на грумминге
Figma release frame
   ↓ чек-лист дизайнера
Jira: статус "Ready for dev"
   ↓ разработка
Storybook (новые компоненты)
   ↓ design review на staging-профиле
Production (за фича-флагом)
   ↓ A/B-тест или прямой релиз — решение PO
Все клиенты
```

## Ответственные

- **[Виктория](/team#виктория) (Product Designer)** — основной автор макетов, проводит ревью кода.
- **[Елена](/team#елена) (Designer Lead)** — критика дизайна раз в две недели, апрув значимых решений.
- **[Анна](/team#анна)** — иногда подключается на конкретные экраны рантайма, если у [Виктории](/team#виктория) много нагрузки (по согласованию).

## Чек-лист дизайнера перед хендоффом

### Структура файла

- [ ] Release frame с именем `<feature> / vX.Y`.
- [ ] Внутри — фрейм «Specs» с текстовой постановкой и ссылкой на Jira.
- [ ] Все слои названы осмысленно.
- [ ] Использованы компоненты из `Forge / Components`. Локальных дубликатов нет.

### Состояния (для редактора и рантайма раздельно)

- [ ] Default / hover / focus / active / disabled.
- [ ] Loading (если применимо).
- [ ] Empty state (для списков, таблиц, пустых разделов).
- [ ] Error (что показывает пользователь при сбое API).
- [ ] В редакторе: состояние **selected** и **invalid** для полей.
- [ ] В рантайме: состояние **submitting** и **submitted** для финального экрана.

### Адаптивность

- [ ] Редактор — `md` и `lg`. Если экран должен поддерживать `xl` — отдельный фрейм.
- [ ] Рантайм — `xs`, `sm`, `md`. Без проверки на мобильном фрейме хендофф не закрывается.

### Контент

- [ ] Тексты — реалистичные, на русском (Forge сейчас один язык).
- [ ] Минимум два варианта длины контента (короткий / длинный).
- [ ] Иконки — Lucide или из набора `forge-icons` (см. [Стайлгайд](/projects/forge/styleguide)).

### A/B и фича-флаги

- [ ] Если задумывается A/B — указаны вариант A и вариант B как отдельные фреймы.
- [ ] Указано имя фича-флага, согласованное с PO.
- [ ] Описано поведение для пользователей вне эксперимента (default — старый вариант).

## Что приходит разработчику

В Jira-задаче:

1. Ссылка на release frame (не на файл целиком).
2. Acceptance criteria.
3. Список затронутых компонентов из дизайн-системы.
4. Имя фича-флага.
5. План A/B (если применимо) или «прямой релиз».

## Критерии приёмки разработчиком

Разработчик возвращает задачу в «Need design», если:

- состояние не покрыто (типичный пример — забыто пустое состояние);
- использован локальный стиль вместо токена;
- размеры на разных брейкпоинтах не согласованы;
- макет ссылается на компонент, которого нет в [Дизайн-система](/projects/forge/design-system) и который не помечен как новый.

## Дизайн-ревью кода

После реализации — [Виктория](/team#виктория) проверяет на staging:

1. На одном из пилотных профилей (см. [Процессы](/projects/forge/processes)).
2. По чек-листу: размеры, отступы, состояния, поведение.
3. Если всё ок — `design-review: ok` в Jira (часть DoD).
4. Замечания фиксирует комментариями к PR со ссылкой на конкретный фрейм Figma.

## A/B-тестирование

Forge — единственный из трёх проектов студии, в котором A/B-тесты — рутина. Логика:

- эксперимент включается через LaunchDarkly с rollout 50/50 на пилотных клиентах;
- метрика — заранее объявленная (обычно одна из бизнес-метрик из [Паспорта проекта](/projects/forge/passport));
- срок — 2–3 недели, но не более одного спринта на принятие решения;
- по результату [Виктория](/team#виктория) и [Наталья](/team#наталья) пишут краткое решение в Confluence и фиксируют, какой вариант остаётся.

## Версионирование макетов

Release frame не редактируется после хендоффа. Любая правка — новая версия (`v1.1`, `v2.0`). Молча менять переданный фрейм запрещено.

> ⚠️ Внимание: фича-флаг — не индульгенция. Если фича перестаёт быть приоритетной и флаг отключается, дизайнер удаляет release frame из активных и помечает его как `archived`. Это критично, чтобы не возвращаться к устаревшему дизайну позже.
