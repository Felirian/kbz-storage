## Специфика Forge

### Next.js (App Router)

SSR используется для:

- публичной страницы шаблонной галереи (SEO);
- страниц помощи и документации (MDX-контент);
- авторизованных страниц редактора и runtime-формы — с гидрацией.

Серверные компоненты (RSC) применяются ограниченно — только для статических участков. Редактор и runtime-форма — клиентские, потому что критически зависят от стейта и DnD.

### Zustand + Immer

Состояние редактора схемы — глубоко вложенный объект (секции → группы → поля → опции). Прямая работа со spread'ами становится неуправляемой, поэтому Zustand-стор обернут middleware Immer:

```ts
const useEditorStore = create(
  immer<EditorState>((set) => ({
    schema: emptySchema(),
    addField: (sectionId, field) =>
      set((state) => {
        state.schema.sections.find(s => s.id === sectionId)?.fields.push(field);
      }),
  }))
);
```

> ⚠️ Внимание: Immer аккуратно работает с массивами, но имеет известные ловушки при работе с `Map`/`Set`. В Forge принято хранить коллекции как массивы с уникальным `id`. Подробнее — в [Архитектура приложения](/projects/forge/architecture).

### React DnD

DnD-конструктор использует HTML5 backend в desktop-сценариях. Для тач-устройств подключается `dnd-multi-backend` (тач-fallback). Все типы перетаскиваемых элементов описаны в `src/shared/dnd/types.ts`.

### JSON Schema

Каждая собранная форма сериализуется в **расширенную JSON Schema**: стандартная схема описывает валидацию, а в `x-forge-*` extension-полях хранятся UI-настройки (раскладка, варианты отображения, условная видимость).

Валидация на runtime — Ajv с предсобранным валидатором (компилируется один раз на загрузку формы).

## Обоснование ключевых выборов

**Next.js, а не Vite SPA.** Forge имеет публичные SEO-чувствительные страницы (галерея шаблонов, документация). Без SSR это потребовало бы отдельного сайта — лишние сложности. Поэтому Next.js покрывает оба сценария.

**Zustand + Immer вместо Redux Toolkit + Immer.** RTK выглядит более «энтерпрайзно», но boilerplate'а в нём больше, а Forge сейчас оптимизируется по скорости итерации. После Q2 (стабилизация) команда переоценит решение.

**Ajv против Zod.** Команда оценивала Zod как удобную DSL, но Forge сериализует схемы для хранения и обмена с бэкендом — нужен стандартизированный формат. JSON Schema этому соответствует, Zod — нет.

**React DnD против dnd-kit.** Решение унаследовано с этапа MVP. Миграция на dnd-kit обсуждается, но не приоритет (см. техдолг в [Беклог и roadmap](/projects/forge/backlog-roadmap)).
