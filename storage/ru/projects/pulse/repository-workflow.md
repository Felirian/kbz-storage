# Работа с репозиторием Pulse

## Branching-стратегия: GitFlow

Pulse — production-проект с предсказуемым релизным окном, поэтому используется классический GitFlow.

```
main          ── релизные теги (v1.x.y), всегда совпадает с production
  └─ release/* ── стабилизация перед релизом
       └─ develop ── интеграционная ветка
            ├─ feature/PULSE-1234-описание
            ├─ bugfix/PULSE-2345-описание
hotfix/*    ── от main, для срочных правок production
```

Правила:

- ветки от `develop`, имя — `<type>/<JIRA-ID>-<краткое-описание-латиницей>`;
- `release/*` создаётся в среду второй недели спринта (см. [Процессы](/projects/pulse/processes));
- `hotfix/*` создаётся от `main` и после мерджа возвращается и в `main`, и в `develop`;
- прямой push в `develop` и `main` запрещён настройкой репозитория.

## Конвенция коммитов

Используем **Conventional Commits**.

```
<type>(<scope>): <subject>

<body — что и зачем, опционально>

<footer — Refs/Closes ссылки>
```

Допустимые типы: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `chore`, `build`, `ci`.

Примеры:

```
feat(dashboard): add turnover prediction widget
fix(employees): correct pagination total on filter reset
perf(charts): debounce realtime updates to 300ms
```

Один логический коммит = одно изменение. Перед мерджем PR через `Squash and merge` собирается финальное сообщение по этому шаблону.

## Pull Request

### Создание

PR из `feature/*` → `develop`. Шаблон PR:

```markdown
## Что
Краткое описание изменений.

## Зачем
Ссылка на Jira, контекст, бизнес-причина.

## Как проверить
Шаги для ревьюера.

## Чек-лист
- [ ] CI зелёный
- [ ] Тесты добавлены/обновлены
- [ ] Документация обновлена
- [ ] Скриншоты UI приложены (если применимо)
- [ ] Нет breaking changes API
```

### Размер

Целевой объём PR — до 400 строк изменений. PR крупнее 800 строк должен быть разбит, либо требует отдельного pre-review с Кириллом.

### Ревью

- минимум один апрув от Middle ([Кирилл](/team#кирилл)) — обязателен;
- архитектурные изменения требуют дополнительного апрува от Dev Lead ([Андрей](/team#андрей));
- джун может ревьюить как читатель, без права апрува;
- срок ответа на запрос ревью — 1 рабочий день.

### Чек-лист ревьюера

1. Соответствует ли код [Код-стайл](/projects/pulse/code-style)?
2. Нет ли глубоких импортов между сегментами FSD ([Архитектура приложения](/projects/pulse/architecture))?
3. Покрытие тестами не упало?
4. Нет ли утечек состояния между компонентами?
5. Запросы к API соответствуют конвенциям ([API-взаимодействие](/projects/pulse/api-integration))?
6. Для UI: скриншоты приложены, поведение совпадает с макетом, есть отметка дизайнера?
7. Производительность: нет ненужных ре-рендеров, тяжёлые вычисления мемоизированы?

## CI

GitHub Actions гоняет на каждый PR в `develop` и `release/*`:

| Шаг | Время (среднее) |
|---|---|
| `npm ci` (с кэшем) | 40 сек |
| ESLint | 30 сек |
| Type-check (`tsc --noEmit`) | 50 сек |
| Vitest (с покрытием) | 2 мин |
| Playwright smoke | 3 мин |
| Bundle size check | 15 сек |
| Vercel preview deploy | 1.5 мин |

Все шаги обязательны для мерджа.

## Релиз

1. PM создаёт `release/x.y.0` от `develop` в среду второй недели.
2. На `release/*` допускаются только багфиксы и обновление версии.
3. После demo (пятница) `release/*` мерджится в `main` и обратно в `develop`.
4. На `main` ставится тег `vX.Y.0`, тег триггерит деплой на AWS.
5. Релиз-нот публикуется в Confluence и в `#pulse-client`.

> ⚠️ Внимание: при hotfix'е сначала мерджим в `main`, ставим тег, деплоим — и только потом мерджим в `develop`. Обратный порядок может привести к потере фикса в production.
