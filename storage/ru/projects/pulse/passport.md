# Паспорт проекта Pulse

## Общая информация

**Название:** Pulse
**Тип системы:** SaaS-платформа для HR-аналитики
**Статус:** production, активная поддержка
**Цель:** предоставить HR-отделам единый дашборд для отслеживания метрик по сотрудникам, текучести и вовлечённости в режиме, близком к реальному времени.

Pulse внедрён у нескольких корпоративных клиентов и работает в режиме платной подписки. Продуктовая команда сосредоточена на стабильности, расширении набора метрик и оптимизации производительности отчётов.

## Заказчик и пользователи

Заказчиком выступают HR-директора и руководители отделов персонала компаний-клиентов. Основные пользователи — HR-аналитики, которые работают с дашбордами ежедневно, и руководители подразделений, которым нужны сводки по своим командам.

Ключевые ожидания заказчика:

- актуальность данных не реже одной минуты от события в источнике;
- предсказуемое время отклика интерфейса при работе с массивами в десятки тысяч записей;
- возможность подключить корпоративный SSO и выгружать отчёты в формате XLSX/CSV.

## Бизнес-метрики

Команда отслеживает четыре продуктовые метрики:

| Метрика | Целевое значение | Текущее |
|---|---|---|
| MAU (активные HR-пользователи) | 4 500 | 3 920 |
| Retention (M3) | ≥ 80% | 83% |
| Time-to-insight (медиана) | < 15 сек | 18 сек |
| NPS | ≥ 40 | 37 |

Подробности по приоритизации улучшений — в разделе [Беклог и roadmap](/projects/pulse/backlog-roadmap).

## Технологический контекст

Стек проекта основан на общем стеке студии (React 18, TypeScript, SCSS Modules, Zustand) и расширен Recharts для визуализаций, React Query для серверного кэша и WebSocket для real-time обновлений. Полный перечень — в [Используемые технологии](/projects/pulse/tech-stack).

## Ключевые даты

- Q1 2025 — релиз модуля «Текучесть»
- Q2 2025 — переход на новую схему агрегации (см. [Архитектура приложения](/projects/pulse/architecture))
- Q3 2025 — публичный API для интеграции с корпоративными BI-системами
- Q4 2025 — пересмотр SLA по доступности (целевой uptime 99.9%)

## Риски

> ⚠️ Внимание: основной операционный риск — пиковая нагрузка на WebSocket-канал в момент массовой выгрузки данных из HR-источника заказчика. Меры по митигации описаны в разделе [API-взаимодействие](/projects/pulse/api-integration).

Второй заметный риск — зависимость от схемы данных конкретного HRIS-клиента. Любое изменение модели на стороне источника может потребовать миграции коннектора.
