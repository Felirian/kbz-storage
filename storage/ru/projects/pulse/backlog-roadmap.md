# Беклог и roadmap Pulse

## Принципы приоритизации

Pulse использует фреймворк **RICE** для оценки задач продуктового беклога. Каждая инициатива размером больше эпика получает оценку:

```
RICE = (Reach × Impact × Confidence) / Effort
```

- **Reach** — сколько пользователей коснётся изменение в течение квартала (в людях, не в %);
- **Impact** — вклад в продуктовые метрики из [Паспорта проекта](/projects/pulse/passport): 3 — массивный, 2 — высокий, 1 — средний, 0.5 — низкий, 0.25 — минимальный;
- **Confidence** — уверенность в оценке (100% / 80% / 50%);
- **Effort** — оценка в человеко-неделях.

Для багов и техдолга RICE не применяется — они сортируются по Severity (см. [Процессы](/projects/pulse/processes)) и техническому риску.

Оперативный беклог багов ведётся отдельной доской в Jira — `PULSE-BUG`.

## Roadmap

### Q2 2025 (текущий)

- релиз материализованных агрегаций;
- бета-доступ к прогнозу текучести для двух пилотных клиентов;
- снижение time-to-insight до 12 секунд (медиана).

### Q3 2025

- публичный read-only API для интеграции с BI-системами;
- двухфакторная аутентификация для административного раздела;
- редизайн раздела «Сотрудники» (силами [Олега](/team#олег), см. [Дизайн-система](/projects/pulse/design-system)).

### Q4 2025

- SLA-апгрейд: целевой uptime 99.9%, расчёт через внешний синтетический мониторинг;
- A/B-тестирование вариантов главного экрана (инструментарий — Statsig);
- завершение миграции с Recharts на собственную обёртку поверх D3 для критичных графиков (RFC уже принят, см. [Архитектура приложения](/projects/pulse/architecture)).

## Критерии перевода задачи в спринт

Задача готова к включению в спринт, если выполнено **всё**:

- описана в Jira по шаблону «Контекст / Задача / Acceptance criteria»;
- есть RICE-оценка (для продуктовых) или Severity (для багов);
- если требует UI — приложены ссылки на Figma и подтверждение от Олега;
- если требует архитектурного решения — приложен RFC;
- эстимация согласована командой на груминге.

## Технический долг

Раздел `tech-debt` в Jira ведётся отдельно. Команда выделяет до 20% ёмкости каждого спринта на работу с долгом. Список верхнеуровневых пунктов:

1. Замена устаревшей версии `react-query@4` на TanStack Query v5.
2. Унификация кастомных хуков для WebSocket — сейчас три разных реализации.
3. Покрытие e2e-тестами критичных пользовательских сценариев (Playwright, см. [Используемые технологии](/projects/pulse/tech-stack)).
