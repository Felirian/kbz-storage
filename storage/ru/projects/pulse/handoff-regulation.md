# Регламент передачи макетов Pulse

## Цель

Регламент описывает, в каком виде дизайнер передаёт макеты в разработку, какие проверки выполняет до передачи, и по каким критериям разработчик принимает работу. Цель — исключить расхождения между Figma и кодом и сократить время сборки фичи.

## Поток передачи

```
Figma (working file)
   ↓ ревью у Designer Lead
Figma (release frame)
   ↓ хендофф-чек-лист
Jira (вложение, ссылка на frame)
   ↓ разработка
Storybook (новые/изменённые компоненты)
   ↓ design review
Production
```

Промежуточные итерации остаются в working file и не передаются в разработку. Передача — только из release frame.

## Чек-лист дизайнера перед хендоффом

[Олег](/team#олег) (или другой дизайнер) проходит этот список до того, как ставит задачу в статус «Ready for dev» в Jira.

### Структура файла

- [ ] Макет в release frame с именем формата `<feature> / vX.Y`.
- [ ] В release frame есть фрейм «Specs» с краткой текстовой постановкой.
- [ ] Все слои названы осмысленно (`button-primary`, не `Rectangle 27`).
- [ ] Используются компоненты из библиотеки `Pulse / Components`, а не локальные дубликаты.

### Состояния

- [ ] Для интерактивных элементов показаны default / hover / focus / active / disabled (см. [Дизайн-система](/projects/pulse/design-system)).
- [ ] Загрузка: skeleton или спиннер.
- [ ] Пустые состояния: что показывается, если нет данных.
- [ ] Ошибки: что видит пользователь при сбое запроса.

### Адаптивность

- [ ] Сделаны ключевые брейкпоинты `md` и `lg` (минимум).
- [ ] Если экран должен работать на `sm` — фрейм для `sm` тоже приложен.
- [ ] Указано поведение длинных текстов (truncate / wrap).

### Токены

- [ ] Все цвета и шрифты — из `Pulse / Tokens`. Локальные стили не используются.
- [ ] Отступы кратны 4px.

### Контент

- [ ] Тексты — реалистичные, не lorem ipsum. Для метрик — правдоподобные числа.
- [ ] Длинный/короткий контент: показано минимум два варианта.
- [ ] Иконки — из набора Lucide (см. [Стайлгайд](/projects/pulse/styleguide)).

### Спецификации

- [ ] Указаны нестандартные анимации и их длительность.
- [ ] Указано поведение при overflow таблиц / графиков.
- [ ] Указано, что считается ошибкой формы и текст сообщений.

## Что приходит разработчику

В Jira-задаче в момент передачи должно быть:

1. Ссылка на конкретный release frame в Figma (а не на файл целиком).
2. Список затронутых компонентов из дизайн-системы (новых или изменённых).
3. Acceptance criteria (что считаем «готово»).
4. Поведение при ошибках API (какие тексты показываются).

## Критерии приёмки разработчиком

Разработчик имеет право вернуть задачу в статус «Need design» если:

- макет не покрывает один из обязательных пунктов чек-листа;
- состояние не описано (например, отсутствует пустое состояние таблицы);
- использованы локальные стили вместо токенов;
- макет ссылается на компонент, которого нет в дизайн-системе и который не помечен как новый.

## Дизайн-ревью кода

После реализации фичи [Олег](/team#олег) делает design review:

1. Проверяет на staging-окружении (а не по скриншотам).
2. Сверяет с release frame по чек-листу: размеры, отступы, состояния, поведение.
3. Если всё ок — ставит в Jira метку `design-review: ok` (это часть Definition of Done, см. [Процессы](/projects/pulse/processes)).
4. Замечания фиксирует комментариями к PR со ссылками на конкретные элементы Figma.

## Версионирование макетов

При значительных изменениях release frame **не редактируется**, создаётся новая версия (`v1.1`, `v2.0`). Старая остаётся для сверки и истории.

> ⚠️ Внимание: если правки в макете прилетели после старта разработки — это change request, оформляется через PM (см. [Процессы](/projects/pulse/processes)). Молча обновлять release frame нельзя.
