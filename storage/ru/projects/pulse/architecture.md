# Архитектура приложения Pulse

## Подход

Pulse построен по облегчённой версии **Feature-Sliced Design (FSD)**. Из канонической схемы используются три верхних слоя — `app`, `pages`, `widgets`, `features`, `entities`, `shared`. Слой `processes` не выделяется: бизнес-флоу укладываются в `features`.

## Структура папок

```
src/
  app/              # инициализация: провайдеры, роутер, глобальные стили
  pages/            # роуты верхнего уровня
    dashboard/
    employees/
    turnover/
    settings/
  widgets/          # композитные блоки, специфичные для страниц
    metric-card/
    employees-table/
  features/         # бизнес-фичи (use-cases)
    filter-employees/
    export-report/
    realtime-sync/
  entities/         # доменные сущности и их базовые UI
    employee/
    department/
    metric/
  shared/           # переиспользуемое
    ui/             # дизайн-система, обёртки над низкоуровневыми компонентами
    lib/            # утилиты, хуки, обёртки над библиотеками
    api/            # axios-instance, базовые клиенты
    config/         # константы, env
```

## Паттерны декомпозиции

**Выбор слоя.** Переиспользуемый UI без знания о домене → `shared/ui`. Представление доменной сущности → `entities`. Пользовательский сценарий → `features` (или `widgets`, если речь о сборке UI для страницы).

**Container / Presenter.** Внутри `widgets` и `pages` явно разделяем компоненты:
- `*.container.tsx` — работа с данными (React Query, Zustand, side-effects);
- `*.view.tsx` — чистый UI, принимает props.

Это упрощает тестирование (`view` тестируется снапшотами без моков сети) и ревью. Не дробим, если работы с данными нет или компонент элементарный (`Button`, `Badge`).

**Извлечение в хуки.** Когда контейнер разрастается, выносим логику в кастомные `use*`-хуки рядом с компонентом или в `shared/lib/` (если переиспользуется). Контейнер становится координатором хуков, а не свалкой `useEffect`.

**Compound components.** Для коробочных UI-компонентов с вариативным составом (`Modal.Header` / `Modal.Body` / `Modal.Footer`, `Tabs`, `Accordion`) — вместо десятка пропсов-флагов. Применяем в `shared/ui`; внутри фич обычно избыточно.

**Public API сегмента.** Каждая папка слоя экспортирует только то, что нужно снаружи, через `index.ts`. Импорт по глубокому пути (`features/x/internal/y`) запрещён ESLint-правилом — см. [Код-стайл](/projects/pulse/code-style).

**Запрет горизонтальных импортов.** Внутри слоя `features` фичи не импортируют друг друга. Если нужна общая логика — она поднимается в `entities` или `shared`.

**View не знает о сторах.** Если внутри `*.view.tsx` появился `useStore` или хук данных — граница нарушена, переносим в контейнер.

## Управление состоянием

| Тип состояния | Решение |
|---|---|
| Серверный кэш | React Query |
| Глобальный UI-стейт (тема, фильтры дашборда) | Zustand |
| Локальный (формы, переключатели в карточке) | `useState` / `useReducer` |
| Производный | селекторы Zustand + `useMemo` |

Глобальные стора живут в `src/app/store/` и собираются по доменам. Каждый стор экспортирует `use<Name>Store` и набор селекторов.

## Real-time

WebSocket-клиент (`src/shared/lib/realtime/`) поднимает соединение в `app`-слое и публикует события в EventEmitter. Фичи, нуждающиеся в апдейтах, подписываются точечно. Конкретные правила — в [API-взаимодействие](/projects/pulse/api-integration).

## Сборка и роутинг

Vite в режиме `splitChunks` режет бандл по страницам. Точка входа — `src/app/main.tsx`. Роутер собирается лениво: каждая страница — `lazy(() => import(...))`.

```ts
const Dashboard = lazy(() => import('@/pages/dashboard'));
```

Бандл по странице держим в пределах 250 КБ (gzipped). Превышение блокирует мердж — проверка в CI.

## RFC-процесс

Любое изменение, затрагивающее структуру слоёв, подключение новой библиотеки или замену существующей, требует RFC. Шаблон в Confluence:

1. Контекст и проблема.
2. Альтернативы (минимум две).
3. Предлагаемое решение.
4. Влияние на производительность, бандл, обратную совместимость.
5. План миграции и rollback.

RFC обсуждается на техсинке (см. [Процессы](/projects/pulse/processes)) и апрувится Dev Lead.

## Производительность

Базовые ограничения:

- TTI на главном дашборде — не более 2.5 секунды на эталонном клиенте (4G, среднее устройство);
- ре-рендер карточки метрики при WebSocket-апдейте укладывается в 1 кадр (16мс);
- React Query devtools отключены в production-сборке (`import.meta.env.PROD`).

> ⚠️ Внимание: при добавлении нового глобального провайдера в `app/` обязательно проверяйте бандл-анализатором, что не подтянулась лишняя библиотека.
