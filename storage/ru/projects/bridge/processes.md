# Процессы проекта Bridge

## Спринт-цикл

Bridge работает в двухнедельных спринтах. До релиза цикл синхронизирован с релизами публичного API на бэкенде — спринт Bridge заканчивается через неделю после релиза API, чтобы успеть обновить документацию и протестировать sandbox.

| День | Активность |
|---|---|
| Пн (нед. 1) | Планирование (90 мин) |
| Чт (нед. 1) | Бэкенд-синк (30 мин), security-ревью открытых задач |
| Ежедневно | Стендап 11:30 (15 мин) |
| Ср (нед. 2) | Pre-release freeze, regression на sandbox |
| Пт (нед. 2) | Демо для заказчика и одного партнёра, ретро |

## Ритуалы

### Планирование

90 минут. Ведёт [Андрей](/team#андрей). Команда фиксирует скоуп спринта, выделяет задачи под security-ревью и документационные.

### Бэкенд-синк

Раз в неделю. Цель — синхронизироваться по изменениям OpenAPI-спецификации. Если в спринте бэкенда меняется контракт публичного API, Bridge должен это отразить в документации и playground'е.

### Security-ревью

Не отдельный ритуал, а отметка в задаче. Любая задача, помеченная `security: yes`, проходит дополнительный ревью у Андрея до мерджа.

### Демо

С заказчиком (отдел партнёрств) и опционально одним из пилотных партнёров. Длительность — 45 минут.

### Ретро

«Start / Stop / Continue», 45 минут.

## Definition of Done

Задача считается завершённой, если:

- код смерджен в `develop` через PR (Bridge на feature-branch, см. [Работа с репозиторием](/projects/bridge/repository-workflow));
- покрытие изменённых файлов юнит-тестами не упало;
- если задача `security: yes` — security-ревью пройдено и зафиксировано в Jira;
- задача проверена на staging-пилоте Андреем или Кириллом;
- если есть UI — [Анна](/team#анна) поставила `design-review: ok`;
- документация обновлена (это критично для Bridge — документация **является** продуктом);
- релиз-нот добавлен в Confluence (если фича пользовательская).

## Управление инцидентами

Bridge ещё не в production, но в пилоте действует SLA на пилотных партнёров:

| Severity | Описание | Реакция | До фикса |
|---|---|---|---|
| S1 | Sandbox недоступен / нельзя выпустить ключи | 1 час | 1 рабочий день |
| S2 | Не работает playground для одного из методов | 4 часа | 2 рабочих дня |
| S3 | Документационная неточность, обходимый баг | 1 день | 1 спринт |
| S4 | Косметика | бэклог | по приоритету |

После релиза SLA будет ужесточён — обсуждение запланировано на Q3 (см. [Беклог и roadmap](/projects/bridge/backlog-roadmap)).

Дежурство по инцидентам — [Кирилл](/team#кирилл) и [Марина](/team#марина) по неделе. [Андрей](/team#андрей) — эскалация.

## Управление изменениями

Любое изменение скоупа спринта оформляется как change request. На пред-релизной фазе у Bridge консервативный подход: добавления в спринт почти не происходят, отклонения предпочитают сдвинуть на следующий спринт.

Архитектурные и security-изменения требуют RFC. RFC обязателен для:

- любого изменения в обработке OAuth-токенов;
- изменения политики хранения ключей партнёров;
- введения новой внешней зависимости в sandbox;
- любого изменения, которое может расширить attack surface.

Шаблон RFC и процесс — в [Архитектура приложения](/projects/bridge/architecture).

## Синхронизация документации

Документация Bridge — два слоя:

1. **Generated** — собирается из OpenAPI-спецификации, обновляется автоматически при релизе API. Команда не редактирует эти файлы вручную.
2. **Conceptual** — гайды, туториалы, объяснения. Пишутся командой в MDX. Обновляются вручную.

Раз в спринт Марина проходит чек-лист синхронизации:

- все методы из OpenAPI отражены;
- удалённые методы помечены как `deprecated`;
- conceptual-гайды не ссылаются на удалённые методы.

> ⚠️ Внимание: рассинхрон между Generated и Conceptual — самая частая категория багов в Bridge. Это причина выделенного чек-листа.
