# Регламент передачи макетов Bridge

## Цель

Bridge — pre-launch проект с высокими требованиями к деталям UI документации и playground'а. Регламент описывает, в каком виде [Анна](/team#анна) (UX/UI Designer) передаёт макеты в разработку, какие проверки выполняет до передачи, и по каким критериям разработчик принимает работу.

## Поток передачи

```
Figma working file
   ↓ критика у Designer Lead ([Елена](/team#елена)), 1 раз / 2 недели
Figma release frame
   ↓ чек-лист дизайнера
Jira: статус "Ready for dev"
   ↓ разработка
Storybook (новые/изменённые компоненты)
   ↓ design review на staging
Production (после релиза)
```

Особенность Bridge: для документационных компонентов добавляется **проверка в живом MDX** — [Анна](/team#анна) обязательно смотрит, как компонент выглядит в окружении настоящих абзацев документа, а не только в изоляции Storybook.

## Чек-лист дизайнера перед хендоффом

### Структура файла

- [ ] Release frame с именем `<feature> / vX.Y`.
- [ ] Внутри — фрейм «Specs» с текстовой постановкой и ссылкой на Jira.
- [ ] Все слои названы осмысленно.
- [ ] Использованы компоненты из `Bridge / Components`. Локальных дубликатов нет.

### Состояния

- [ ] Default / hover / focus / active / disabled.
- [ ] Loading (если применимо).
- [ ] Empty state.
- [ ] Error state.
- [ ] Состояния для дублей в **тёмной теме** (обязательно).

### Адаптивность

- [ ] Документационные компоненты — `xs`, `sm`, `md`, `lg`. Все четыре фрейма приложены.
- [ ] Sandbox-компоненты — `md`, `lg`.
- [ ] Указано поведение длинных идентификаторов / кодов / URL: truncation или wrap.

### Тёмная тема

- [ ] Каждое состояние имеет вариант для тёмной темы.
- [ ] Контраст текста на фоне проверен (target — WCAG AA).
- [ ] Используются токены из `Bridge / Tokens` (никаких HEX напрямую).

### Документационные компоненты — особое

- [ ] Компонент показан в потоке абзацев (минимум один фрейм с реальным абзацем сверху и снизу).
- [ ] Соблюдён вертикальный ритм с типографикой.
- [ ] Проверена интеграция с `CodeBlock` / `Callout`, если применимо.

### Контент

- [ ] Тексты — реалистичные, на русском (Bridge сейчас один язык; локализация — Q1 2026, см. [Беклог и roadmap](/projects/bridge/backlog-roadmap)).
- [ ] Длинный/короткий контент: показаны оба варианта.
- [ ] Иконки — Lucide или из набора методов HTTP (см. [Стайлгайд](/projects/bridge/styleguide)).

## Что приходит разработчику

В Jira-задаче:

1. Ссылка на release frame (не на файл целиком).
2. Acceptance criteria.
3. Список затронутых компонентов из дизайн-системы.
4. Инструкции по тёмной теме (если она требует особого поведения, не покрытого токенами).
5. Если компонент будет использоваться в MDX — пример использования.

## Критерии приёмки разработчиком

Разработчик возвращает задачу в «Need design», если:

- состояние не покрыто (особенно — ошибка или dark variant);
- использован локальный стиль вместо токена;
- макет ссылается на компонент, которого нет в [Дизайн-система](/projects/bridge/design-system) и который не помечен как новый;
- документационный компонент не показан в потоке текста.

## Дизайн-ревью кода

После реализации [Анна](/team#анна) проверяет:

1. На staging-окружении (а не только по скриншотам).
2. Для документационных компонентов — на нескольких реальных страницах документации, чтобы увидеть в потоке текста.
3. Обязательно — в тёмной теме.
4. По чек-листу: размеры, отступы, состояния, поведение, контрасты.
5. Если всё ок — `design-review: ok` в Jira (часть DoD, см. [Процессы](/projects/bridge/processes)).

Замечания фиксирует комментариями к PR со ссылкой на конкретные элементы Figma.

## Версионирование макетов

Release frame не редактируется после хендоффа. Любая правка — новая версия (`v1.1`, `v2.0`).

## Сертификационные требования

Bridge готовится к SOC2-аудиту. Это означает, что на стадии хендоффа дизайнер обязательно учитывает:

- доступность по WCAG 2.2 AA;
- видимый focus-ring у всех интерактивных компонентов;
- осмысленные `aria-label` для иконочных кнопок ([Анна](/team#анна) указывает их в спецификации, не оставляет на разработчика).

> ⚠️ Внимание: accessibility — не отдельная фаза, а часть исходного макета. Если [Анна](/team#анна) забывает указать `aria-label` или контраст не проходит — задача возвращается в дизайн до начала разработки.
