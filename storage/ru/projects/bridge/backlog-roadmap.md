# Беклог и roadmap Bridge

## Принципы приоритизации

До релиза Bridge использует **MoSCoW** в чистом виде — без RICE, потому что объём аудитории до запуска не позволяет считать осмысленные Reach. После релиза метод будет дополнен RICE.

- **Must** — без этого релиз нельзя считать выпущенным;
- **Should** — есть в плане релиза, но при критическом отставании может быть перенесено;
- **Could** — после релиза;
- **Won't (this release)** — явно за рамками первого выпуска.

Дополнительный фактор приоритизации, не отражённый в MoSCoW — **security-вес**. Любая задача с `security: yes` поднимается выше других в своей категории. Решение Андрея.

## К релизу (Q3 2025)

### Must

| Инициатива | Статус |
|---|---|
| Управление API-ключами (создание, ротация, отзыв) | в работе |
| Sandbox с симуляцией ответов API (5 ключевых методов) | в работе |
| Документация по OAuth 2.0 + PKCE flow | дизайн |
| Поиск по документации (Algolia DocSearch) | беклог |
| Аудит security: external pentest перед релизом | запланирован Q3 |

### Should

| Инициатива |
|---|
| Multi-tenant: партнёр-организация, роли в организации |
| Виджет статуса публичного API (uptime последних 24ч) |
| Code samples в документации на 3 языках (curl, JS, Python) |

### Could

- Dark mode портала;
- Подписка на changelog API по e-mail;
- Сертификаты соответствия (SOC2 — на странице Security).

### Won't (this release)

- Биллинг и тарифы партнёров (Q4 2025);
- Marketplace внешних интеграций;
- Локализация документации.

- маркетплейс готовых интеграций;
- локализация документации (en).

## Критерии перевода задачи в спринт

Чтобы задача попала в спринт, должно быть выполнено всё:

- категория MoSCoW проставлена и не Won't;
- описание содержит «Контекст / Задача / Acceptance»;
- определено, требует ли security-ревью (поле `security: yes/no`);
- если требует UI — макет утверждён Анной (см. [Регламент передачи макетов](/projects/bridge/handoff-regulation));
- если меняется поведение OAuth или ключей — приложен RFC.

## Технический долг

Сейчас долг минимален — проект молодой. Зафиксированные пункты:

1. Полное покрытие e2e-тестами sandbox-сценариев (Playwright).
2. Автоматизация проверки актуальности conceptual-гайдов (проверять при сборке OpenAPI, что упомянутые методы существуют).
3. Замена внутренней реализации tabbed code samples на готовый компонент.

> ⚠️ Внимание: пункт 2 заблокирован до стабилизации формата OpenAPI у бэкенда. Команда бэкенда планирует переход на 3.1.0 в Q3.
