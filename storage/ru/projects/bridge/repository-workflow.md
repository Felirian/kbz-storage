# Работа с репозиторием Bridge

## Branching-стратегия: feature-branch

Bridge придерживается классического **feature-branch workflow**. Это срединный вариант между GitFlow ([Pulse](/projects/pulse/repository-workflow)) и trunk-based ([Forge](/projects/forge/repository-workflow)). Выбор обоснован:

- продукт ещё не в production, но требования к качеству высокие (security);
- отсутствие фича-флагов LaunchDarkly как массовой практики (Bridge их использует точечно);
- релизный темп — раз в спринт, не чаще.

```
main         ── production-ready (после релиза); сейчас совпадает с staging
develop      ── интеграционная ветка
  ├─ feature/BRIDGE-1234-...
  ├─ fix/BRIDGE-2345-...
  └─ chore/BRIDGE-3456-...
```

Правила:

- ветки от `develop`, имя — `<type>/<JIRA-ID>-<кратко-латиницей>`;
- срок жизни ветки — 3–5 рабочих дней; долгие задачи декомпозируются;
- прямой push в `develop` и `main` запрещён;
- `main` обновляется только при релизе из `develop` через PR.

## Конвенция коммитов

**Conventional Commits**:

```
feat(sandbox): add JSON body autocomplete from OpenAPI
fix(auth): correct PKCE verifier storage cleanup on logout
docs(guides): rewrite quickstart with PKCE flow
chore(deps): bump swagger-ui-react to 5.11.0
```

Допустимые типы: `feat`, `fix`, `refactor`, `perf`, `test`, `docs`, `chore`, `build`, `ci`, `security`. Тип `security` — специфика Bridge: используется для коммитов, прямо повышающих защищённость продукта (без подробного описания уязвимости).

Сообщения на английском. PR мерджится через `Squash and merge`.

## Pull Request

### Создание

PR из feature-ветки → `develop`. Шаблон PR:

```markdown
## Что
Краткое описание изменений.

## Зачем
Ссылка на Jira, контекст.

## Security
- [ ] Задача `security: yes`? (если да — указано отдельно)
- [ ] Не введены новые внешние зависимости в auth/sandbox?
- [ ] Не появились localStorage/sessionStorage в новых местах?

## Документация
- [ ] Conceptual-гайды обновлены (если применимо)
- [ ] Generated-reference синхронен с OpenAPI

## Чек-лист
- [ ] CI зелёный
- [ ] Тесты добавлены
- [ ] Скриншоты UI приложены
- [ ] Lighthouse Score не упал на ключевых страницах
```

### Размер

Цель — PR до 500 строк изменений. Документационные PR могут быть больше — они оцениваются отдельно.

### Ревью

- минимум один апрув от Senior ([Марина](/team#марина)) или Dev Lead ([Андрей](/team#андрей));
- задачи `security: yes` — обязательный апрув от [Андрея](/team#андрей), независимо от других;
- задачи, затрагивающие OAuth или ключи — апрув [Андрея](/team#андрей) + второй ревьюер ([Марина](/team#марина));
- срок ответа на запрос ревью — рабочий день.

### Чек-лист ревьюера

1. Соответствие [Код-стайл](/projects/bridge/code-style).
2. Архитектурные границы доменов (`domains/<x>` не импортирует другие домены напрямую).
3. Тип `security: yes`? — пройден security-чек-лист (см. ниже).
4. Внешние данные типизированы через сгенерированные типы.
5. Документация согласована с кодом.
6. UI: скриншоты приложены, есть отметка дизайнера.
7. Бандл-чек пройден (см. [Архитектура приложения](/projects/bridge/architecture)).
8. E2E-сценарии не сломаны.

### Security-чек-лист (для `security: yes`)

1. Не вводятся ли новые места хранения чувствительных данных?
2. Не расширяется ли CSP-allowlist? (если да — обоснование в PR)
3. Не появляются ли новые внешние HTTP-вызовы из браузера?
4. Корректна ли обработка ошибок: не утечёт ли информация о структуре системы?
5. Логирование не содержит токены / ключи / PII?

## CI

GitHub Actions, на каждый PR в `develop`:

| Шаг | Среднее время |
|---|---|
| `npm ci` (с кэшем) | 50 сек |
| ESLint + security-плагин | 40 сек |
| Type-check | 1 мин |
| Vitest (с покрытием) | 2 мин |
| Playwright smoke | 4 мин |
| Bundle size check | 30 сек |
| Lighthouse CI на ключевых страницах | 3 мин |
| OpenAPI sync check | 30 сек |
| Vercel preview deploy | 1.5 мин |

Все шаги обязательны для мерджа.

## Релизы

Релиз — мердж `develop` → `main`. Запускается еженедельно (ритуал) или по необходимости (при критическом фиксе).

Шаги:

1. [Андрей](/team#андрей) создаёт PR `develop → main` с changelog'ом.
2. CI прогоняется ещё раз на актуальной `develop`.
3. После апрува PR мерджится, на `main` ставится тег `vX.Y.Z`.
4. Тег триггерит деплой на S3 + CloudFront invalidation.
5. Algolia crawler перезапускается.
6. [Сергей](/team#сергей) публикует release notes в Confluence и канале партнёрств.

## Hotfix

Hotfix — короткая ветка от `main` (`hotfix/BRIDGE-...`), мерджится сначала в `main`, затем в `develop`. Hotfix-PR обязательно содержит:

- описание инцидента;
- security-чек-лист (по умолчанию);
- план верификации.

> ⚠️ Внимание: в Bridge запрещён `--no-verify` коммит и любой обход pre-commit-хуков. Это требование [Андрея](/team#андрей) и закреплено в правилах ревью.
